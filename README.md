# Treasure Hunter Solution

This repository contains a solution for the Leadvise Reply Treasure Hunter challenge. The original challenge statement is kept in `README..md`; this file documents the submitted approach in `solution.js`. The time spent on the challenge was approximately four hours, covering brainstorming, prompt engineering, experimentation with different approaches, and analyzing the outputs generated by Codex across both low- and high-level aspects.


## Challenge Understanding

The simulator calls `new Stacker().turn(cell)` once per game cycle. The agent sees only the current tile and the four direct neighbors, then returns one action: `"left"`, `"up"`, `"right"`, `"down"`, `"pickup"`, or `"drop"`.

The goal is to stand on the gold tile at height 8. Because the agent can only move across a height difference of 1, it must collect blocks and build a staircase ending next to the gold tile.

Important mechanic: `pickup` and `drop` happen on the current tile. Picking up a block lowers both the tile and the agent by 1; dropping raises both by 1. This means the staircase must be built layer by layer so the agent never creates a height gap it cannot cross.

## Approaches Considered

| Approach | Summary | Tradeoff |
| --- | --- | --- |
| Robust explorer | Build a relative map from local observations, find gold, choose a safe staircase path, gather blocks, and build by layers. | Most reliable and used in `solution.js`. |
| Quick heuristic | Wander until gold is found, collect nearby blocks opportunistically, and greedily build around the tower. | Shorter to write, but brittle and likely to waste turns. |
| Simulator-aware | Tune around known generator constants such as an 18x18 board, 50 walls, 50 blocks, and height 8. | Fast, but less general and more coupled to this exact challenge file. |

## Chosen Algorithm

`solution.js` implements the robust explorer:

1. Keep a relative coordinate system and update known tiles from each `turn(cell)` call.
2. Explore reachable frontier cells with BFS until the gold is discovered and enough block supply is known.
3. Plan an 8-cell staircase path whose base has access to enough non-stair block sources.
4. Reserve the staircase cells so the agent does not consume its own ramp as block supply.
5. Build by layers: raise cells `1..7` to level 1, cells `2..7` to level 2, and so on.
6. Before taking a source block, simulate the pickup height change and verify that the agent can still reach the current build target.
7. Climb to the last step and move onto the gold tile.

The solution uses plain browser JavaScript only. It does not read canvas state, DOM state, or private variables from `challenge.js`.

Height 12 fails with this map setup because 50 total blocks are available while the minimum required staircase mass is higher.
Even in the optimistic case where each staircase cell already starts at level 1, a height-12 target needs at least 55 drops.

For the height-8 target, both `Robust Explorer` and `Balanced Fast` achieved `100/100` runs at or below 1000 turns in this benchmark.

The browser `challenge.html` ultrafast runner should be used as the final submission benchmark because it is the official environment. The local mirror was used during development to make repeated 100-run checks scriptable.

## Running

Open `challenge.html` in a browser, uncheck Test Mode, enable Ultrafast, and run 100 trials. The simulator automatically loads `solution.js`.

To benchmark all implemented methods locally:

```bash
node scripts/benchmark_methods.js --heights 8 --runs 100 --cycle-limit 30000
node scripts/benchmark_methods.js --heights 7,10 --runs 60 --cycle-limit 30000
node scripts/benchmark_methods.js --heights 12 --runs 20 --cycle-limit 10000
```
